# PRD: review-gym

**作成者**: Ryusei | **日付**: 2026-03-29 | **ステータス**: ドラフト
**アイデアID**: 693 | **ドメイン**: プログラミング教育/メンタリング | **難易度**: Medium

## 問題定義

コードレビューは開発者にとって最も重要なスキルの一つだが、体系的に練習する手段がない。Exercism や HackerRank はコーディング力を鍛えるが、「他人のコードを読み、問題を見抜く力」を鍛えるプラットフォームは存在しない。実際の OSS プロジェクトの過去 PR（匿名化済み）を題材に、セキュリティ脆弱性・パフォーマンス問題・設計上の懸念を見つけ出す「目利き力」を定量的に鍛えるトレーニング環境が必要。

## 目標と成功指標

| 目標 | 指標 | ターゲット |
|------|------|----------|
| レビュー練習の継続利用 | WAU（週間アクティブユーザー） | MVP後1ヶ月で50 WAU |
| レビュースキルの可視化 | ユーザーのスコア改善率 | 平均20%向上（10セッション後） |
| 練習問題の充実 | 収録PR数 | MVP時点で30問以上 |
| 弱点分析の有効性 | 推薦PRの完了率 | 70%以上 |
| 開発者コミュニティでの認知 | GitHub Stars | 3ヶ月で100 Stars |

## スコープ外

- リアルタイムのペアレビュー機能
- 自動PR収集パイプライン（MVPでは手動キュレーション）
- 課金・サブスクリプション機能
- モバイルネイティブアプリ
- AI による自動レビュー生成（模範解答は人手でキュレーション）
- チーム管理・企業向け機能

## ユーザーストーリー

- **ジュニア開発者**として、実際のOSSのPR差分を見てレビューコメントを書きたい。なぜなら実践的なレビュースキルを安全な環境で練習できるから。
- **ジュニア開発者**として、自分のレビューと模範解答を比較してスコアを確認したい。なぜなら自分のレビュー力がどの程度か客観的に知りたいから。
- **中級開発者**として、セキュリティ・パフォーマンス・設計など弱点カテゴリを分析してもらいたい。なぜなら効率的にスキルギャップを埋めたいから。
- **中級開発者**として、弱点に基づいた練習PRを推薦してほしい。なぜなら限られた時間で最も効果的な練習をしたいから。
- **シニア開発者**として、自分のレビュー観点を共有してコミュニティに貢献したい。なぜなら後進育成に興味があるから。

## 技術要件

### 技術スタック

- **バックエンド**: Go（API サーバー）
- **フロントエンド**: TypeScript / React
- **データベース**: PostgreSQL
- **キャッシュ**: Redis
- **インフラ**: Docker Compose（開発環境）

### アーキテクチャ

モノレポ構成:
- `backend/` - Go API サーバー（REST）
- `frontend/` - React SPA（Vite）
- PostgreSQL でPRデータ・ユーザー進捗・スコアを永続化
- Redis でセッション管理・レビュー進捗のキャッシュ

### 非機能要件

- **パフォーマンス**: API レスポンス < 200ms（p95）、差分表示の初回ロード < 1秒
- **セキュリティ**: PR データの匿名化保証、ユーザー認証（GitHub OAuth）
- **スケーラビリティ**: 同時接続100ユーザーまでを初期目標

## 競合分析

| 競合 | URL | ギャップ |
|------|-----|---------|
| Exercism | https://exercism.org | コーディング練習のみ。レビュー練習機能なし |
| HackerRank | https://hackerrank.com | アルゴリズム・面接対策中心。レビュースキルは対象外 |
| CodeReview StackExchange | https://codereview.stackexchange.com | Q&A形式。体系的なトレーニングではない |

**差別化ポイント**: 実際のOSS PRを題材にしたコードレビュー練習はExercism等のコーディング練習と方向が異なる独自コンセプト。コードレビュー練習プラットフォームは皆無。セキュリティ脆弱性/パフォーマンス問題の目利き力訓練は独自。

## 技術リスク

| リスク | 発生可能性 | 影響 | 軽減策 |
|--------|----------|------|--------|
| OSS PR の匿名化が不完全 | 中 | 高（プライバシー問題） | 匿名化パイプラインのテスト強化、手動レビュー併用 |
| 模範レビューのキュレーションコスト | 高 | 中（コンテンツ不足） | MVP は30問に絞り、テンプレート化で効率化 |
| diff 表示の UX が複雑 | 中 | 中（ユーザー離脱） | 既存の diff ライブラリ活用、段階的に改善 |
| スコアリングの妥当性 | 中 | 中（信頼性低下） | 複数のスコアリング軸、ユーザーフィードバック反映 |

## マイルストーン

### Phase 0: MVP（2週間）

1. **基盤セットアップ**（2日）: CI/CD、リンター、テストフレームワーク、DB スキーマ
2. **コアデータモデル**（2日）: PR・レビューコメント・スコアのスキーマ設計、シードデータ
3. **差分表示 + レビュー入力 UI**（3日）: 匿名化 PR の差分表示、行単位のコメント入力
4. **スコアリングエンジン**（3日）: 模範レビューとの比較、精度・網羅性スコア算出
5. **弱点分析 + 推薦**（2日）: カテゴリ別スコア集計、重点練習 PR の推薦ロジック
6. **統合テスト + ドキュメント**（2日）: E2E テスト、README、デモ

### Phase 1: Growth（MVP 後 1ヶ月）

- GitHub OAuth 認証
- ユーザーダッシュボード（スコア推移グラフ）
- PR 問題の難易度レベル分け
- コミュニティ機能（レビュー共有）
